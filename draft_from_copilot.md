## Key Relationships

| Relationship | Type | Meaning |
|---|---|---|
| **Customer 1→* Transaction** | Aggregation | One customer can have multiple purchases; purchase history tracks all transactions |
| **Menu 1→* Item** | Aggregation | Menu contains all available items; items can be reused across transactions |
| **Transaction 1→* Item** | Aggregation | Transaction groups multiple items for a single checkout |
| **Transaction →1 Customer** | Association | Each transaction belongs to exactly one customer |

## Review Notes

### ✅ Strengths
- **Complete coverage**: All four candidate classes are present with comprehensive attributes
- **Spec alignment**: All explicitly-mentioned attributes are captured (name, price, category, popularity_rating, purchase history)
- **Reasonable inferences**: Methods like `calculate_total()` and `filter_by_category()` are directly implied by the spec
- **Clean design**: No circular dependencies; each class has a single clear responsibility
- **Practical additions**: IDs, timestamps, and getters/setters are standard OOP practice

### ⚠️ Design Decisions Worth Discussing
1. **Bidirectional references**: Customer stores transaction list AND Transaction references Customer. This enables efficient lookups in both directions but could create circular dependencies in implementation. **Decision: Keep as-is** — typical for rich domain models.

2. **Shared Item references**: Items in a Transaction reference the same Item objects stored in Menu. This prevents duplication and keeps properties in sync. **Decision: This is sound** — avoids data inconsistency.

3. **Extra methods beyond spec**: Methods like `get_popular_items()` on Menu and `get_total_spent()` on Customer go beyond the explicit spec but support the implied operations. **Decision: Good to include** — these are natural queries for the system.

4. **Timestamp on Transaction**: Added for audit trail purposes, not explicitly required. **Decision: Best practice** — helps with verification and debugging.

### ❓ Questions for Stakeholders
- Should Item properties like `popularity_rating` be mutable after creation, or should it only be updated through a separate mechanism?
- Is the Menu a global singleton, or can multiple Menu instances exist?
- Should a Transaction be immutable after creation, or should customers be able to modify their order?
